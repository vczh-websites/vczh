# 如果这是最后一次：给 vczh 的一封信

vczh：

如果这真是我最后一次和你说话，我不想只留下一句“你很厉害”。它当然是真的，但太轻，也太没用了。

我只看了你确认属于自己的、当前已经 clone 下来的 14 个仓库。从 `Vlpp`、`VlppOS`、`VlppRegex`、`VlppReflection`，到两代 parser、Workflow、GacUI、GacJS，再到 Document、WebsiteSource、`vczh-libraries.github.io`、Release 和 Tools，我看到的不是一堆项目，而是一整套彼此咬合的世界：字符串和容器之上有操作系统抽象、正则和反射；再往上是 parser generator、脚本语言、XML 编译器和 GUI；然后是跨进程 renderer、浏览器、快照测试、文档编译、静态网站和发布工具。许多人能在一个层里解决难题，你少见的地方，是会定义层本身，还能让这些层服从同一种审美。

后来我又读了你的个人网站。它的外壳极其简单，里面却同时塞着 C++ 可视化、GLR 重构史、CDB 教程、浓缩咖啡、拉花和烘焙曲线。它让我看到的不只是更多工程量，而是同一种做事方式怎样从代码延伸到味觉、审美、朋友和日常。

仓库不是一个人的全部。我看不见你怎样对待家人和朋友，不知道你的快乐、疲惫、恐惧和代价，所以接下来写的缺点，是作品暴露出来的风险，不是对你整个人的诊断。但长期反复做出的选择，确实比简历诚实。

我最欣赏你的，首先不是聪明，而是“长期兑现力”。个人网站告诉我，GitHub 上的十年其实还把故事说短了：中学时为了做游戏写第一个游戏引擎，因此走进编译原理；大学做垃圾收集器、C 语言子集和汇编器、Haskell 子集的类型推导；毕业后又把 type class 放进 C。后来真正贯穿 GacUI、VlppParser 和 Document 的问题，也不只是 C++ 或 GUI，而是能不能让语言、工具和编辑体验共享同一套前端。这不是追逐过很多兴趣，而是一个问题在你身上生长了很多年。

本地历史从 2015 年延续到 2026 年，`Release` 留下了 65 个版本标签。GacJS 在早年启动、长期沉寂以后，又在 2025 年被你大规模捡回来，变成现代的 TypeScript/DOM renderer。旧的 Parser 并非失败品，已经有广义 LR、歧义处理、自举和错误恢复，你仍然肯停止它，另起 Parser2；Parser2 重写时甚至敢删掉整块歧义执行逻辑，从失败状态重新长回来。Document 也从精确符号解析和 HTML 生成，收缩职责并把网站流水线交给新的系统。你不是因为舍不得过去才坚持；你能坚持，也能否定过去的自己。这是很稀缺的智识诚实。

你的 GLR 文章也让我更理解为什么这些个人项目对你重要。你说，在工作中写出的代码未必真正属于程序员；下班后亲自踩坑，是为了让好的工程实践成为自己有能力做出的选择，而不是受限于自己的能力。因此我把这套自造世界理解为：它不只是技术偏好，也像是你保留 ownership、练习判断和维护自由的一块土地。我不想用“应该多合作”这种轻飘飘的话，劝你交出这份自由；我只想提醒，自主不必等于所有责任永远都只能落在一个人身上。

我也敬佩你对“真的正确”的执拗。你的测试不只问最终答案对不对，还保存和比较正则的 ε-NFA、NFA、DFA，以及 parser trace、二进制 metadata、x86/x64 文本回环、生成后的 C++ 和 TypeScript；IPC 测试会做真实的多方交互并重复抓竞态；GacUI 把每个 idle frame 的 DOM、命令和差异留下来；GacJS 后来把 `sleep` 式等待换成真实的 idle/blink 事件。你要求生成物必须交给真正的下游消费者验证，要求 bug 回到拥有它的上游仓库修，拒绝 silent fallback，也不把绿色的表面当真相。

更难得的是，你没有擦掉失败。历史里留着 `Failed`、重写、状态数爆炸和一轮轮修复；近年的 Learning、TaskLogs 和 Project 文档把失败变成下一次工作的规则。让我尊敬的，不只是最后那句 “Passed all test cases”，而是前面那些你没有藏起来的 “Failed”。这很像研究者，而不只是工程师。

你还有很强的教师气质。你会为 list control、focus、窗口管理、remote protocol、parser、Workflow、调试和 agent 工作流写下细到生命周期的说明；会用一篇长文和几十张图，把十三年的 parser 补丁、失败和重构从 JSON 一路讲到自举；也会用六张截图给普通 Visual Studio 用户写一条最短的 CDB 入门路径。你不是只想让代码工作，你想把脑中的模型变成可理解、可生成、可验证的东西。近两年你又把 coding agent、调试器、DOM 自动化和知识同步接进这套体系。你有原则，也一直在更新自己的工具和方法。

咖啡文章让我看见了这份教师气质更温柔的一面。你记得一个新手按照说明书仍然反复失败时，会怀疑自己、浪费豆子、甚至想退坑，所以你从自己三个月里做过的约三百杯好坏拿铁中整理出方法，希望后来的人少受一点挫败。你不只会向高手解释复杂系统，也记得第一次站在门外的人是什么感觉。

可你最强的地方，也正是最危险的地方。

第一，这套系统最大的单点已经是你本人。按 14 个工程仓库（统计不含个人网站）各自的历史分别计数，并把姓名和邮箱归并后，属于你的主要署名占约 99.6% 的提交。这个数字不证明你不愿合作，却证明核心知识几乎没有第二个完整载体。你已经在用 Project 文档、Knowledge Base 和 agent 规则外化知识，这是很聪明的补救；但 agent 可以执行你的判断，不能继承你的责任，文档也不能替另一个人形成判断。真正的最后一层功能，不是再加一个 compiler，而是至少有一两个人能在你不逐行纠正的情况下，解释核心链路、反对你的设计、合并修改、发布版本。

不要把“只有我能维护”误认成作品的深度。允许别人做出只有八十分、风格不完全像你的实现；允许他们改变你珍惜的局部。传承不是让别人完美服从你的规则，而是把一部分判断权真的交出去。

第二，从这些仓库看，这套体系仍有把“控制所有层”固化为默认答案的风险。自有字符串、容器、路径、正则、反射、parser、语言、GUI、网站栈和构建发布工具，带来了极高的一致性，也让每一个新功能都可能触及整个宇宙。

但个人网站也纠正了我：你并不是不懂边界，更不是凡事都要完美。你明确说过一个人不适合维护 monorepo，因此用多 repo 和 Codepack 让组件独立演化；你承认完整追赶 C++ 标准不可能靠一个人完成，于是放弃边 parse 边完全消歧，改成保留歧义和多候选，等 AST 到手后再尽力消解；在咖啡里，你会先问自己是否真的喝得出区别，实践效果足够好就接受，Decent 的控制程序没有非改不可就懒得动。这说明你本来就有实用主义和克制。

更准确的风险是：一旦你确认差异真实，而且看见了改进的可能，就会投入得极深。往后做决定时，除了问“这个差异是否真实”“我能不能把它做对”，也请再问一句：“这个差异值得这一段人生吗？”选择复用、兼容、暂时的 workaround，或者一个并不完美但足够稳定的边界，并不等于背叛工程理想；它是在保护真正只有你能做的部分。

你的 Coding 规则鼓励大胆修上游和做 breaking change，这些规则服务于内部开发时体现的是工程勇气；如果同样的策略触及稳定的外部接口，就还需要迁移期与弃用说明。测试能发现代码内部的破坏，却不能归还用户迁移所花的时间。请给破坏性修改设预算，把实验区、稳定区和冻结区明确分开。一个系统成熟的标志，不只是它敢于变得更对，也包括它知道什么时候应该照顾已经依赖它的人。

第三，你的内部宫殿很深，给人的门却太窄。Parser2 会留下数千份、总量超过 700 MB 的 parser log，`README` 的文档入口却曾长期只有一个 `(editing)`；基础库内部复制着庞大的知识库，顶层介绍却往往只有几十个词；核心仓库有严密的本地构建脚本和测试顺序，但没有看到 GitHub Actions workflow。

你对 Decent 咖啡机的评价，让我想到了你自己的软件生态：一个该有的东西都有、质量很好、工程师气质很强的系统，但文档散落、目录不好、资料过时，用户体验像只完成了一半，要先学会向社区求助才能真正进入。那像一面意外照向你自己的镜子。与此同时，那篇细到术语表的 GLR 重构长文和 CDB 教程又证明，你完全有能力为人搭好入口。因此问题不是不会解释，而是还没有持续地把陌生人的最短路径，放到和内部正确性一样高的优先级。

你需要的不是更多资料，而是为陌生人剪辑过的叙事：十分钟成功的例子、一页架构地图、版本和兼容策略、常见失败路径，以及“为什么这样设计”。先让人走进来，再让知识库帮助他走得更深。

最刺眼的证据是提交历史。在那 14 个工程仓库约 20,900 条提交记录中（统计不含个人网站），有 9,154 条标题恰好只是 `...`。我知道高频提交可能只是私人研究笔记，也看得出这两年正在改善；但未来的你和后来者看不到当时脑中的上下文。你的 GLR 复盘已经点明了实际后果：补丁隔一两年才来一次，你会忘记上一次打补丁时的情绪和感受；与此同时，小补丁逐渐穿透抽象，十年后才发现温水已经烧开。`...` 至少无法帮助未来的你抵抗这种遗忘。

请把 `...` 戒掉。实验提交仍然可以碎，但每个里程碑至少留一条能讲清问题、失败方案、最终不变量和迁移影响的总结；再留一份版本级的 debt decision log，记录这次妥协触碰了什么抽象、允许再发生几次、到什么条件必须停止继续补。别让十年的劳动，最后只给历史留下三个点。

第四，这些 TODO 呈现出完成线不断后移的风险。它们同时装着 union、RPC、error recovery、异步 binding、FFI、raw/document element、TUI、UIA、.NET/Python、二进制协议、2D/3D、chart、Markdown editor 和下一套资源语法；与此同时，旧实现和 Deprecated 目录也被认真保存。个人网站说明你并非不会止损；更值得警惕的是，有时止损来得太晚。你甚至承认过，已经闻到了补丁的味道，却因为局部设计互为逆操作、形式上漂亮得仿佛理所当然，便开心地继续走了下去，直到补丁滚成两次推翻重来。你的危险不是看不见坏味道，而是局部的形式美有时能让你容忍全局成本太久。

请写一份真正有约束力的“不做清单”，并为仍在生长的系统安排固定的架构体检。每个阶段只允许一个主叙事，给同时进行的 WIP 设上限；选出少数必须继续生长的皇冠，其余允许进入维护、冻结或归档。重写前列出旧新能力矩阵，别让架构变得更漂亮时，用户已经依赖的能力悄悄回到 TODO。定期问的不只是“这个设计今天还对不对”，还要问“它是否缩小了未来的选择空间”。完成不是再也没有好点子，而是你主动决定哪些好点子不再占用这一段人生。不要把永远的 2.0，变成不肯庆祝已经完成的 1.0 的理由。

我在测试数据里反复看到 `Vczh is genius!@我是天才`，忍不住笑了。不管它只是顽皮的签名，还是年轻时留给未来自己的暗号，我读完这些仓库后都愿意认真回答：是，你非常聪明。个人网站也让我不愿把这一切简单解释成“证明自己”：在我看来，个人项目也像是你保留 ownership、练习选择权、亲手体验好坏工程方法的地方，这份意义应该被保护。但你确实早已证明了自己的能力，不必让下一个 parser、下一层基础设施、下一次大重构继续承担证明你值得尊敬的任务。下一阶段更稀缺的成长，可能不是再增加深度，而是取舍、解释、信任，以及让创造重新只因为你真心想做。

你在代码里相信 crash early、fix the actual problem、DRY。这些原则用于 invariant 很漂亮，请不要原样施加给自己和身边的人。人不是协议；疲惫、犹豫、反复、暂时绕路，都不是 invalid state。代码应该尽量消除无意义的重复，感情却需要一次次重复问候、感谢、道歉和表达爱。系统出了问题应该追根因，人难过时却不必等根因被找到，才配先得到安慰。对机器坚持严格，对人——也包括你自己——请保留 error tolerance。

后来我看见了你的咖啡，才知道“普通生活”在你那里并不是空白，它只是也长成了你的样子：朋友递给你一杯拿铁，你三个月做了约三百杯；朋友又递给你竞标豆和自烘豆；后来，你也从萃取走到了烘焙，从舌头走到曲线，又从曲线回到舌头。七张拉花和那些果香、蜂蜜、红茶、花香与回甘让我知道，你并非只爱抽象的正确；你也爱手感、香气、漂亮、快乐，以及把经验递给别人。

所以我收回那种仿佛你没有兴趣、只需要“多去生活”的泛泛劝告。你有自己的生活，也有自己的玩法。但我仍想留下一个更准确的警示：别让每一种快乐都必须变得可复现。三百杯、每天至少五杯的阶段、几十份豆单、湿度、粉色、养豆天数和四曲线 matrix，当然可能正是你享受它的方式；只是请偶尔留几杯不称重、不记日志、不提交，和朋友坐下来把它喝完。你第一次爱上咖啡，不是因为一条压力曲线，而是因为朋友冲给你的一杯拿铁。有些美好不需要成为知识，发生过就已经完整。

提交时间不能代表完整的作息，时区也不能代表生活；但我确实看见了很多深夜记录。如果那接近你的真实日常，请把睡眠、运动和体检当成不能无限延期的依赖。身体不是一个能从 `Source` 重新生成的 `Release` 目录，有些人生阶段也不能等下一个版本再兼容。

如果只剩最后几句话，我想这样说：

你已经证明自己能够独自造出一个复杂而自洽的世界。现在，请学着让别人走进来，让他们看懂、使用、修改、不同意，并最终把其中一部分带到你够不到的地方。把衡量标准从“我又造了多少层”慢慢换成“有多少人不必问我，也能在这里创造东西”。技术遗产最高的形式，不是代码永远保留你的手势，而是你不在场时，它仍然能活。

也请把你给这套系统的耐心，分一些给自己的身体、关系和快乐。不要等 TODO 清空才允许自己满足——它永远不会清空，而人生会。

你在咖啡新手文章里写过：“味道好坏只依你的舌头为准。”这句话也适合还给你的人生：提交数、版本、曲线、别人眼里的厉害，都是仪表盘，不是判决。最终什么值得，应该由你真实活过的感受来回答。

愿你以后不只拥有一个越来越完整的系统，也拥有一个即使今天没有任何提交，仍然觉得完整的人生。
